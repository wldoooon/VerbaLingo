# Research: Implement Batch Sync Logic

**Date**: 2026-02-06

## 1. Executive Summary
This research outlines the logic for the "Janitor" background task that synchronizes usage from Redis to PostgreSQL in batches.

## 2. Technical Context
- **Affected File**: `app/services/usage_service.py`
- **Database Engine**: `app.db.session.engine` (AsyncEngine).
- **Redis Set**: `usage:dirty_users`.

## 3. Findings & Analysis
- **Atomicity**: Using `redis.spop("usage:dirty_users", count=100)` allows us to grab a batch of users to process while others continue to be added to the set by incoming requests.
- **Session Management**: Background tasks need to manage their own `AsyncSession` lifecycle using the global `engine`.
- **Sync Strategy**:
    - For each dirty user, we fetch their current daily counts from Redis.
    - We fetch their `UserUsage` record from Postgres.
    - We update the daily counts in the DB to match Redis.
    - We update the lifetime counts based on the difference (or we can just keep lifetime counts in the DB and increment them based on what was synced).
- **Concurrency**: Since the scheduler runs every 5 minutes, we must ensure that a slow sync doesn't overlap with the next one (APScheduler handles this with `max_instances=1`).

## 4. Technical Constraints
- **Asynchronous Execution**: The background task is synchronous by default in `BackgroundScheduler`, but we can use `AsyncIOScheduler` or just run the async function in the event loop.
- **SQLModel**: We'll use `session.get(UserUsage, user_id)` for quick retrieval.

## 5. Architecture Documentation
The sync logic will be a top-level function in `usage_service.py` that can be imported into `main.py` for the scheduler.
