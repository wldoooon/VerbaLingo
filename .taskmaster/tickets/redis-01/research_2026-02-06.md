# Research: Implement Redis Dirty Set Tracking

**Date**: 2026-02-06

## 1. Executive Summary
This research documents the current state of usage incrementing in the `UsageService` and identifies the exact points where Redis "Dirty Set" tracking should be injected.

## 2. Technical Context
- **Affected File**: `app/services/usage_service.py`
- **Key Function**: `increment_usage(redis, db, user, feature, client_ip)`
- **Current Behavior**:
    - Generates a Redis key: `usage:{identifier}:{feature}:{today}`
    - Increments the value using `redis.incr()`.
    - Sets a TTL of 48 hours.
- **Data Flow**: The function returns the new count but does not trigger any persistence logic. Persistence is currently handled by `sync_usage_to_db`, which is called elsewhere (likely as a FastAPI `BackgroundTask`).

## 3. Findings & Analysis
- **User Identification**: The code distinguishes between logged-in users (`user:ID`) and anonymous users (`ip:IP`). 
- **Dirty Set Scope**: We only need to sync logged-in users to PostgreSQL. Therefore, the `SADD` operation should only trigger if `user` is not `None`.
- **Atomic Operations**: `redis.sadd` is atomic. Adding it after `redis.incr` ensures we track the user for the next sync cycle.

## 4. Technical Constraints
- **Redis Connection**: The `redis` object passed to the function is an `aioredis` instance. We must use `await redis.sadd()`.
- **Key Name**: The dirty set should have a consistent name, e.g., `usage:dirty_users`.

## 5. Architecture Documentation
The project uses a standard FastAPI service pattern. `UsageService` acts as the coordinator between Redis and the Database.
