# Research: Usage Service Refactor & Consolidation

**Date**: 2026-02-06

## 1. Executive Summary
This research identifies a major architectural fragmentation between `UsageService` and `TieredRateLimiter`. It outlines the strategy to consolidate these into a single "Source of Truth" for feature limits.

## 2. Technical Context
- **System A (`UsageService`)**: Manages counters, Dirty Set tracking, and DB persistence.
- **System B (`TieredRateLimiter`)**: Manages sliding window Redis limits but lacks persistence.
- **Routes**: Use System B via decorators.

## 3. Findings & Analysis
- **Fragmentation**: Currently, a user can hit their limit in System B, but System A (the DB) never hears about it. Conversely, System A has the Dirty Set sync logic, but it's never called by the routes.
- **The Solution**: 
    - Keep `TieredRateLimiter` for **Security** (IP-based sliding windows).
    - Refactor `TieredRateLimiter.check_feature_limit` to be a wrapper around `UsageService.check_usage_limit`.
    - Modify the `@feature_rate_limit` decorator to call `UsageService.increment_usage` upon successful request completion.

## 4. Technical Constraints
- **Dependencies**: The limiter decorator needs access to the `Redis` client and `AsyncSession`. We'll use the existing dependency injection patterns.
- **Async Execution**: Everything is async, so we must `await` all service calls.

## 5. Architecture Documentation
Consolidating these systems ensures that the "Dirty Set" tracking added in `redis-01` actually triggers, as it will now be part of the request lifecycle.
